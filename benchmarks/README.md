# First Technology Bake-off Reproducibility Plan

- Plan ID: `PM-TECH-BAKEOFF-001`
- Observed: 2026-07-18
- Data classification: synthetic only
- Local candidate results: not run
- Production architecture decision: not made

This directory defines the first bounded Private Match technology experiments. It selects research tracks and evidence requirements; it does not report candidate performance, establish a security property, or select a production implementation.

## 1. Evidence and execution boundary

No local candidate benchmark has been executed for this artifact. Timing or scale figures on upstream pages remain vendor-, project-, or author-published observations and must not be copied into a local result.

Every later execution must:

1. use only inputs generated by `generate_inputs.py` or the committed `.test` fixtures;
2. pin the implementation revision, dependency lock, container digest, operating system, CPU, memory, and network shaping;
3. preserve the exact command arguments and exit codes;
4. retain SHA-256 digests of inputs, outputs, stdout, and stderr;
5. use one of `pass`, `fail`, `skip`, `unsupported`, `timeout`, or `tool-error` without translating one status into another;
6. preserve failed and partial artifacts;
7. distinguish a library result from the proposed `MATCH`, `NO_MATCH`, or `INDETERMINATE` wrapper result;
8. state unobserved channels and tooling limits.

Actual result files must validate against
[`result.schema.json`](result.schema.json). Every status requires at least one
input-artifact digest. A generated-input run must bind its input manifest
SHA-256; a fixture run must bind the fixture or fixture-manifest SHA-256. A
`skip` or `unsupported` result must bind the reviewed matrix or input-definition
digest and state in `limitations` that no experiment input was generated. A
`pass` additionally requires at least one output-artifact digest. No result file
is committed by this planning change.

`source_revision` must exactly equal the `source_revision` pinned for the
result's `track_id` in
[`experiment-matrix.yaml`](experiment-matrix.yaml). v0.1 permits no deviation.
A different revision requires a separately reviewed matrix change before
execution; a result file cannot document an after-the-fact exception.

Matrix repository paths are untrusted configuration input to the validator.
Selected technology records, generators, fixtures, and result files must be
relative regular files under their permitted repository prefixes. Absolute
paths, parent segments, missing files, directories, and symlinks that resolve
outside the repository are rejected. Track and experiment identifiers must also
be unique.

## 3. Reference environment record

Before a run, create `benchmarks/work/<run-id>/environment.json` with at least:

- UTC execution time;
- operating-system image and kernel;
- CPU model, logical CPU count, memory, and storage type;
- Docker or other runtime version and image digest;
- compiler, Bazel, Go, Python, and candidate tool versions where applicable;
- source revision and whether the worktree is clean;
- loopback, LAN, or shaped-network topology;
- bandwidth, latency, loss, MTU, and capture interface;
- TEE instance family, enclave CPU/memory allocation, EIF digest, PCRs, debug mode, and attestation verifier revision;
- all deviations from the matrix.

Do not include cloud account IDs, internal hostnames, credentials, keys, private IP inventories, or production topology in a public result.

## 4. Deterministic input generation

Generated material belongs under ignored `benchmarks/work/`, not `/tmp` and not a production data directory.

Example for the small balanced case:

```bash
python benchmarks/generate_inputs.py \
  --output-dir benchmarks/work/exp-001 \
  --case-id exp-001 \
  --left-size 1000 \
  --right-size 1000 \
  --intersection-size 100 \
  --seed 1001
```

The generator treats sizes as distinct identifiers and writes any requested duplicates as additional rows. Identifiers are deterministic `pm-synth-...` values derived from the public seed. The JSON manifest records row counts, distinct counts, duplicate counts, the expected byte-exact distinct intersection, and input digests.

The edge-case fixtures use reserved `.test` addresses:

- [`fixtures/normalization-edge-left.csv`](fixtures/normalization-edge-left.csv)
- [`fixtures/normalization-edge-right.csv`](fixtures/normalization-edge-right.csv)

Expected distinct intersections:

| Profile | Expected | Boundary |
|---|---:|---|
| Byte-exact | 1 | Only the duplicate test identifier is common; duplicate rows do not increase the distinct count |
| Unicode NFKC, trim, and case-fold | 4 | Leading-zero removal is intentionally excluded, so `000123` and `123` remain distinct |

The normalization implementation, version, and owner must be recorded. A candidate that normalizes internally must not be credited with the expected value until its exact behavior is observed.

[`experiment-matrix.yaml`](experiment-matrix.yaml) is the machine-readable plan. It contains all Issue #4 scenarios:

| ID | Required scenario | Primary question |

All matrix entries start at `not-run`. An unavailable public message hook is recorded as `unsupported`, not `pass`. A timeout, tool error, or skipped TEE run remains visible.

## 6. Track procedure — SecretFlow PSI

The selected first protocol is the documented `PROTOCOL_KKRT` two-party path. The SecretFlow documentation identifies KKRT as semi-honest; this plan does not infer malicious-party protection.

Execution preparation:

1. clone the official repository under an approved workspace;
2. checkout commit `d7682707035d6b3e04cc09b8bfef629140641432` without modifying upstream code;
3. record the dependency lock and build tool versions;
4. build the documented optimized target;
5. prepare sender and receiver configurations from the upstream PSI v2 benchmark guide;
6. run each party in an isolated local container or VM with only the generated input mounted read-only;
7. set and record `broadcast_result` explicitly rather than relying on a default;
8. capture each party's output separately and compare distinct-result digests.

The upstream quick start uses privileged containers and host networking. That is an operational risk, not an assurance property. A benchmark host must be disposable, contain no credentials or private data, and record every added capability. A future harness should reduce privileges before candidate adoption; this plan does not silently change the upstream procedure.

The library may return intersection rows. The research wrapper may derive the proposed minimum result for comparison, but must retain the library output as restricted synthetic evidence and must not claim that the upstream API enforces the core output policy.

## 7. Track procedure — RFC 9497 VOPRF through CIRCL

This track verifies the primitive, not set intersection:

1. use CIRCL `v1.6.4` and the `oprf` package;
2. execute RFC 9497 vectors for `ristretto255-SHA512` VOPRF mode;
3. batch the same synthetic inputs used by `EXP-001` and record client/server bytes and elapsed time;
4. bind a public benchmark context to the application transcript outside the primitive and record how the binding is constructed;
5. repeat invalid-element, invalid-proof, wrong-public-key, and stale-context cases;
6. place a 20-query application budget in front of the server for `EXP-006` and show that the primitive itself does not supply this state.

The OPRF server key prevents replacement with a public unkeyed hash, but low-entropy identifiers remain vulnerable to authorized online enumeration and output-oracle effects unless authentication, normalization, purpose binding, rate limits, query budgets, and result policy are supplied by the application. The VOPRF output is client-only and is not a symmetric Private Match result.

## 8. Track procedure — AWS Nitro Enclaves

After explicit human authorization:

1. use an enclave-enabled, supported EC2 parent instance with no customer or production access;
2. install the pinned Nitro CLI and NSM API releases;
3. build a deterministic EIF from reviewed source and record its digest and PCR0/PCR1/PCR2/PCR8 values;
4. run outside debug mode because debug-mode attestation PCRs are zeroed;
5. request an attestation document containing a fresh verifier nonce;
6. validate the AWS Nitro certificate chain, COSE signature, timestamp, nonce, PCR policy, and expected image measurement outside the enclave;
7. send two generated inputs over vsock and allow only the enumerated test result plus a bound receipt to leave the enclave;
8. record parent-visible vsock byte counts, parent network traffic, instance/enclave lifecycle, and every persisted artifact;
9. terminate the enclave and instance and record teardown evidence.

Nitro Enclaves has no direct external network or persistent storage; all application I/O crosses the parent through vsock or an explicitly configured proxy. This reduces the exposed application surface but does not establish dataset authenticity, omission resistance, endpoint safety, availability, traffic-analysis resistance, side-channel resistance, or correct enclave code. The AWS Nitro system and attestation PKI remain trusted components.

## 9. Tamper, replay, and failure handling

For each public candidate interface, the execution PR must identify a serialization boundary before mutation. Mutate only synthetic local traffic:

- truncate one documented message;
- flip one byte outside transport framing and one byte inside a payload where the API permits;
- send an unknown message type or unsupported field;
- resend a message within the same session;
- replay a prior transcript against a new session;
- for TEE, reuse an attestation document with a stale nonce.

Record whether the candidate rejects before producing output, which party observes the error, whether an output file was created, and whether retry changes state. If encrypted transport prevents controlled mutation or the library exposes no supported hook, record `unsupported`. Do not disable verification, TLS, attestation, or integrity protection to force a test to run.

## 10. Packet capture and raw-egress observation

Capture only the isolated synthetic run. The capture host must already grant the capture process the necessary capability; this plan does not require `sudo` or change host security settings.

Collect:

- a packet capture for every IP interface used by the candidate;
- stdout, stderr, trace, temporary, and output files;
- file-open and network endpoint summaries where an approved tracing tool is available;
- parent-side vsock byte counts for the TEE track;
- SHA-256 digests before any artifact is moved.

Search the captured corpus for every raw synthetic identifier and for the documented normalized variants. Report counts by surface. A zero count is limited to the captured artifacts and tools; it does not prove absence from process memory, hardware side channels, peer endpoints, logs outside the harness, or traffic-analysis features.

## 11. Metrics and comparison rules

Measure separately:

- dependency installation and build time;
- key, preprocessing, image, and attestation setup time;
- online wall and CPU time;
- peak resident memory and disk artifacts;
- bytes sent and received per party and per phase;
- output recipient and result symmetry;
- input rows, distinct values, duplicates, normalized values, and expected result;
- retry, timeout, unsupported, and error outcomes;
- operator steps and external services.

Use at least one warm-up and the matrix repetition count. Report all observations rather than only the fastest run. Vendor or author figures may appear only in an attributed background column and never in the local measurement fields.

## 12. Stop conditions

Stop the run and preserve the status if:

- non-synthetic or customer data is encountered;
- a credential, production endpoint, private hostname, or private artifact would enter evidence;
- the source revision or dependency graph cannot be pinned;
- a required integrity or attestation check would need to be disabled;
- an implementation writes raw identifiers outside the declared test directory;
- the query limit or cost ceiling is reached;
- a security-relevant failure needs coordinated disclosure;
- cloud execution lacks explicit approval;
- the tool returns timeout, `unsupported`, or `tool-error`.

## 13. Result review

Before a result is proposed for publication:

- validate its JSON against `result.schema.json`;
- reproduce it from a clean worktree on the recorded environment;
- verify every input and output digest;
- confirm that no generated result contains private data or credentials;
- review license and source provenance without converting that review into patent clearance;
- label candidate, vendor, author, and local results separately;
- include failures, skips, unsupported checks, timeouts, and tooling errors;
- state that experiment selection is not a production PET selection.
